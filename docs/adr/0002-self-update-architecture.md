# 2. Self-Update via Two-Loop Architecture

## Status

Accepted (2026-03-09)

## Context

haniel is a configuration-based Windows service runner. Once installed and registered
as a Windows service, it runs continuously. When haniel's own code is updated on GitHub,
there is no mechanism to apply the update without manual intervention.

The core difficulty: haniel cannot update its own running code. The process that needs
updating is the process that would do the updating. Additionally, as a Windows service
running in Session 0, it cannot prompt the user for confirmation — no GUI, no console,
no interactive input.

### Requirements

1. haniel must detect when its own GitHub repo has new commits
2. Updates must not disrupt managed services without authorization
3. The update mechanism must handle git/pip failures gracefully
4. Users must be able to approve or skip updates (since blind auto-update is risky)
5. The entire mechanism must work non-interactively (Windows service constraint)

## Decision

### Two-Loop Architecture

Introduce a thin PowerShell wrapper script (`haniel-runner.ps1`) between WinSW and
the haniel process. WinSW registers the wrapper as the service, not haniel directly.

```
WinSW Service
  └─ haniel-runner.ps1 (outer loop, ~50 lines)
       ├─ git fetch + reset --hard origin/main
       ├─ pip install -e .
       └─ python -m haniel.cli run haniel.yaml (inner process)
            ├─ starts managed services
            ├─ poll loop (including self-repo)
            └─ exit code signals intent to wrapper
```

### Exit Code Convention

| Exit Code | Meaning                        | Wrapper Action                      |
|-----------|--------------------------------|-------------------------------------|
| 0         | Clean shutdown (`sc stop`)     | Wrapper exits. Service stops.       |
| 10        | Self-update approved           | Wrapper loops: update + restart.    |
| Other     | Crash                          | Wrapper exits. WinSW onfailure.     |

Exit code 10 was chosen because it is outside common error ranges (1-2 generic,
3-5 common app) and below 128 (POSIX signal exits).

### Update Approval Model

When haniel detects changes in its own repo:

- **`auto_update: true`**: Immediately shuts down services and exits with code 10.
- **`auto_update: false`** (default): Sends webhook notification, enters "pending
  approval" state, and exposes `haniel_approve_update()` MCP tool. Update happens
  only when approval is received.

Approval is decoupled from the approval source. haniel only knows "approval arrived
via MCP tool." The caller could be:
- A human using Claude Code interactively
- A Slack bot forwarding a user's button click
- An automated policy (e.g., "approve if tests pass")

This keeps haniel indifferent to the approval mechanism, consistent with its
"intentionally indifferent" design philosophy.

### Wrapper Script Design

The wrapper always runs `git reset --hard origin/main` + `pip install -e .` before
launching haniel. This means:
- First start after reboot gets latest code automatically
- Crash-restart also gets latest code (services already disrupted, so no approval needed)
- `git reset --hard` guarantees local = remote, preventing infinite change detection loops

### Configuration

```yaml
repos:
  haniel:
    url: git@github.com:eiaserinnys/haniel.git
    branch: main
    path: ./.projects/haniel

self:
  repo: haniel          # key from repos section
  auto_update: false    # default: false (opt-in)
```

### Wrapper Configuration

`haniel-runner.conf` (generated by `haniel install`, simple key=value):
```
WEBHOOK_URL=https://hooks.slack.com/services/T.../B.../...
HANIEL_REPO=./.projects/haniel
CONFIG=haniel.yaml
MAX_GIT_FAILURES=3
```

The wrapper reads this file instead of parsing haniel.yaml, keeping it minimal.

### Circuit Breaker

The wrapper tracks consecutive `git fetch` failures. After MAX_GIT_FAILURES (default 3),
it stops trying to update and launches haniel with the currently installed code, plus
sends a webhook notification. This prevents permanent service downtime due to transient
network issues.

## Consequences

### Gained

- Fully automated self-update with user approval gate
- No service disruption without authorization (when `auto_update: false`)
- Crash recovery automatically picks up fixes
- Wrapper is ~50 lines, minimizing its own need for updates
- Approval mechanism is source-agnostic (MCP tool, not tied to Slack or any UI)

### Lost

- Atomic rollback — if an update breaks haniel, recovery requires manual
  `git reset --hard {previous_commit}` + `sc start haniel`
- Wrapper script is in-memory during execution, so wrapper-updating commits need
  two restarts to fully take effect

### Accepted Limitations

**No automatic rollback (v1).**
If new code crashes haniel immediately, WinSW's `<onfailure>` stanza handles
restart with escalating delays. After exhausting retries, the service stops and
a webhook alerts the user. Manual rollback is:
```
git -C ./.projects/haniel reset --hard {previous_commit}
sc start haniel
```
Automatic rollback (tracking previous commit, reverting if startup fails within N seconds)
is deferred to a future version to avoid premature complexity.

**Wrapper script in-memory limitation.**
`git reset --hard` updates the wrapper file on disk, but the currently running
PowerShell process uses its in-memory copy. If a haniel update includes wrapper
changes, the new wrapper takes effect on the next restart. In the worst case
(wrapper change is required for the update to work), two restarts are needed.
This is an extremely rare scenario.

**pip install failure is non-fatal.**
If `pip install -e .` fails (network issue, broken dependency), the wrapper sends
a webhook notification and attempts to launch haniel with the previously installed
code. The old version may still work if the failure was an added (not changed)
dependency.

**"Always update on start" vs "approval gate" is not contradictory.**
The approval gate controls when haniel voluntarily exits (disrupting running services).
The "always update" in the wrapper applies when services are already down (crash,
reboot, approved update). These govern different lifecycle states:
- Services running, code changed → approval needed (haniel decides when to exit)
- Services already down → update automatically (wrapper decides before launch)
